## Design Decisions

### 1. BaseEntity — abstract class, nie TypeORM BaseEntity

TypeORM posiada własną `BaseEntity` (Active Record pattern). Nie używamy jej — nasz projekt stosuje Repository pattern (Data Mapper). Nasza `BaseEntity` to zwykła abstract class z dekoratorami TypeORM, od której dziedziczą encje. Nie importujemy TypeORM `BaseEntity`, więc nie ma konfliktu nazw.

### 2. Board.columns — relacja dodana w feature 003

Spec FR-008 wymaga eager loading relacji `columns`. Ponieważ encja `BoardColumn` nie istnieje jeszcze, relacja `@OneToMany` zostanie dodana w feature 003. W tym feature `findOne()` zwraca board bez tablicy columns.

### 3. Delete — fizyczne usunięcie (MVP)

Konstytucja zezwala na fizyczny delete w fazie MVP (przed wdrożeniem audytu). Board delete jest fizyczny. Soft delete zostanie dodany w przyszłości.
